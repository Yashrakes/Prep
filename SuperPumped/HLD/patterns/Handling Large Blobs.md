# Handling Large Blobs Pattern

## Overview
Pattern for efficiently handling large files (images, videos, documents) in distributed systems. Instead of routing gigabytes through application servers, use direct client-to-storage transfers with presigned URLs and CDN delivery.

---

## When to Use This Pattern
- **Video platforms** - Upload and stream videos (GB-TB files)
- **Photo sharing** - User photos, albums
- **Document management** - PDFs, Office documents, archives
- **Media platforms** - Audio files, podcasts
- **File sharing** - Dropbox-style services
- **Cloud storage** - User file backups
- **Avatar/profile images** - User uploaded images

---

## Why NOT Route Through Application Servers

### Problems with Server-Mediated Upload/Download

**Traditional Approach (BAD)**:
```
Client → Upload 1GB video → App Server → Process → S3

Problems:
1. App server becomes bottleneck
2. High memory usage (buffering entire file)
3. High bandwidth costs
4. Server can crash during transfer
5. Slow upload (client → server → storage = 2x time)
6. Cannot handle concurrent large uploads
```

**Calculation**:
```
1 GB upload via server:
- Client → Server: 1 GB (100 Mbps = 80 seconds)
- Server → S3: 1 GB (1 Gbps = 8 seconds)
- Total: 88 seconds + server processing

10 concurrent uploads:
- Server bandwidth: 10 GB/s needed
- Server memory: 10 GB buffering
- Crashes or becomes unusable
```

**Better Approach**:
```
Client → Presigned URL from server → Upload directly to S3

Benefits:
1. Server only handles metadata (fast)
2. Client uploads directly to storage (parallel, fast)
3. No server bandwidth usage
4. Scales to millions of uploads
```

---

## Solution Architecture

### Core Components

**Upload Flow**:
```
1. Client requests upload permission
2. Server generates presigned URL
3. Client uploads directly to S3 using URL
4. S3 notifies server on completion
5. Server updates database with file metadata
```

**Download Flow**:
```
1. Client requests file
2. Server generates presigned URL (with CDN if available)
3. Client downloads directly from S3/CDN
4. Server logs access (if needed)
```

## Presigned URLs

### What are Presigned URLs?

**Concept**:
- Temporary URL with embedded authentication
- Allows direct access to storage without credentials
- Time-limited and scoped permissions
- Generated by backend with storage credentials

**Example Presigned URL**:
```
https://bucket-name.s3.amazonaws.com/video123.mp4?
AWSAccessKeyId=AKIAIOSFODNN7EXAMPLE&
Expires=1672531200&
Signature=bWq2s1WEIj%2BVGNYiFZMPxZBYqDM%3D
```

**Components**:
- **Resource**: video123.mp4
- **Credentials**: AWSAccessKeyId (embedded)
- **Expiration**: Unix timestamp (e.g., 1 hour from now)
- **Signature**: Cryptographic signature (prevents tampering)

**Key Properties**:
- Anyone with URL can access (temporary public URL)
- Cannot be modified (signature validation fails)
- Expires automatically (time-based security)
- Scoped to specific operation (upload OR download, not both)

---

### Upload with Presigned URLs

**Step-by-Step Flow**:

**Step 1: Client Requests Upload Slot**:
```
Client → POST /api/videos/upload-request
Body: {
  "filename": "birthday.mp4",
  "content_type": "video/mp4",
  "file_size": 1073741824  // 1 GB
}

Server validates:
- User authenticated and authorized
- File size within limits
- Content type allowed
- User has quota available
```

**Step 2: Server Generates Presigned URL**:
```
Server logic:

1. Generate unique file_id (UUID)
2. Construct S3 key: "videos/{user_id}/{file_id}/birthday.mp4"

3. Generate presigned URL (AWS SDK):
   presigned_url = s3.generate_presigned_url(
       ClientMethod='put_object',
       Params={
           'Bucket': 'my-video-bucket',
           'Key': 'videos/user123/uuid/birthday.mp4',
           'ContentType': 'video/mp4'
       },
       ExpiresIn=3600  // 1 hour
   )

4. Create database record:
   INSERT INTO uploads (
       file_id, user_id, filename, status, created_at
   ) VALUES (
       'uuid', 'user123', 'birthday.mp4', 'pending', NOW()
   )

5. Return to client:
   {
       "file_id": "uuid",
       "upload_url": "https://s3.../presigned...",
       "expires_at": "2025-01-15T11:00:00Z"
   }
```

**Step 3: Client Uploads Directly to S3**:
```
Client (browser or app):

fetch(upload_url, {
    method: 'PUT',
    body: file,
    headers: {
        'Content-Type': 'video/mp4'
    }
})

Upload goes directly to S3:
- No application server involvement
- S3 handles authentication (presigned URL)
- Can show progress bar (browser native)
- Can pause/resume (range requests)
```

**Step 4: Server Gets Notified on Completion**:
```
S3 Event Notification → Lambda/SNS → Backend

Backend updates database:
UPDATE uploads 
SET status = 'completed', completed_at = NOW()
WHERE file_id = 'uuid';

Optional post-processing:
- Trigger video transcoding
- Generate thumbnail
- Update user quota
```

### Download with Presigned URLs

**Flow**:

**Step 1: Client Requests File**:
```
Client → GET /api/videos/{file_id}

Server:
1. Check authorization (user can access this file?)
2. Generate presigned download URL:
   
   presigned_url = s3.generate_presigned_url(
       ClientMethod='get_object',
       Params={
           'Bucket': 'my-video-bucket',
           'Key': 'videos/user123/uuid/birthday.mp4'
       },
       ExpiresIn=3600  // 1 hour
   )

3. Return:
   {
       "file_id": "uuid",
       "download_url": "https://s3.../presigned...",
       "expires_at": "2025-01-15T11:00:00Z",
       "content_type": "video/mp4",
       "file_size": 1073741824
   }
```

**Step 2: Client Downloads Directly**:
```
Client redirects or fetches from presigned URL
Download happens directly from S3
Server not involved in data transfer
```

### Presigned URL Security

**Time-Based Expiration**:
```
Short-lived: 15 minutes - 1 hour
- Upload URLs: 1 hour (large files take time)
- Download URLs: 15 minutes (can regenerate)

Benefits:
- Shared URL becomes useless after expiration
- Limits exposure window
```

**Scoped Permissions**:
```
Upload URL: PUT only (cannot download)
Download URL: GET only (cannot upload or delete)

Generated with minimal required permissions
```

**Non-Guessable**:
```
Cryptographic signature based on:
- AWS credentials
- Resource path
- Expiration time

Cannot be forged without credentials
Cannot modify URL (signature validation fails)
```

**Best Practices**:
```
1. Generate new URL for each request (don't reuse)
2. Use shortest expiration practical
3. Validate file after upload (virus scan, format check)
4. Rate limit URL generation (prevent abuse)
5. Log URL generation for audit
```

## Content Delivery Network (CDN)

### Why CDN for Large Files

**Without CDN**:
```
User in Australia → Request image → Origin Server in US
- Distance: ~15,000 km
- Latency: ~200-300ms RTT
- Transfer: 10 MB image over long distance → slow

Every user requests from origin
Origin bandwidth saturated
Expensive egress costs
```

**With CDN**:
```
User in Australia → Request image → CDN Edge in Sydney
- Distance: ~100 km
- Latency: ~10-20ms RTT
- Cached at edge → instant delivery

Origin only serves cache misses
90% requests served from edge
10x faster, 90% less origin load
```

---

### CDN Architecture

**Components**:
```
User → CDN Edge Server (POP) → Origin (S3/Storage)
         ↓ (cache hit)
         Return cached file

         ↓ (cache miss)
         Fetch from origin, cache, return
```

**How It Works**:

**First Request (Cache Miss)**:
```
1. User requests: https://cdn.example.com/images/photo.jpg
2. CDN edge checks cache → miss
3. CDN fetches from origin: https://bucket.s3.amazonaws.com/images/photo.jpg
4. CDN caches file at edge
5. CDN returns file to user
6. Latency: ~200ms (origin fetch)
```

**Subsequent Requests (Cache Hit)**:
```
1. User requests same URL
2. CDN edge checks cache → hit
3. CDN returns cached file immediately
4. Latency: ~20ms (local edge)

10x faster
No origin load
```

---

### CDN Configuration

**Cache-Control Headers**:
```
HTTP Response from origin (S3):

Cache-Control: public, max-age=31536000
-- Cache for 1 year (immutable content)

Cache-Control: public, max-age=86400
-- Cache for 1 day (updated daily)

Cache-Control: private, no-cache
-- Don't cache (user-specific content)

ETag: "abc123"
-- Version identifier for conditional requests
```

**What to Set**:
```
Static assets (images, videos): max-age=31536000 (1 year)
- Use versioned URLs for updates: photo-v2.jpg

User content (profile pics): max-age=86400 (1 day)
- May change, but not frequently

Dynamic content: no-cache
- User-specific data, always fresh
```

---

### CDN with Presigned URLs

**Challenge**:
```
Presigned URLs have query parameters with expiration
CDN should cache underlying file, not expiring URL
```

**Solution 1: Separate URLs**:
```
Origin: Presigned URL (for upload/access control)
CDN: Clean URL (for delivery)

Flow:
1. User uploads to presigned URL → S3
2. File becomes public (or signed with longer expiration)
3. CDN caches file at clean URL: https://cdn.example.com/files/xyz.jpg
4. Users download from CDN (no query params)
```

**Solution 2: Signed Cookies**:
```
Instead of signed URLs, use signed cookies for CDN
Cookie: CloudFront-Policy=...; CloudFront-Signature=...

File URL: Clean https://cdn.example.com/files/xyz.jpg
Auth: Cookie (not in URL)

Benefits:
- Cacheable URLs
- Access control via cookies
```

**Solution 3: CDN Signed URLs** (CloudFront):
```
Generate CloudFront signed URL (separate from S3):
https://cdn.example.com/files/xyz.jpg?Expires=...&Signature=...

CloudFront validates signature
Caches underlying file
Different expiration from S3 presigned URL
```

---

### CDN Purging/Invalidation

**When to Purge**:
```
Content updated:
- New version of image uploaded
- Video replaced
- Need to remove cached content
```

**How to Purge**:

**Invalidation Request** (CloudFront):
```
Invalidation request:
- Paths: ["/images/photo.jpg", "/videos/*"]
- CDN marks cached items as stale
- Next request fetches fresh from origin

Limitation: Can take 5-15 minutes
```

**Versioned URLs** (Recommended):
```
Instead of purging, use versioned URLs:
- Old: https://cdn.example.com/images/photo-v1.jpg
- New: https://cdn.example.com/images/photo-v2.jpg

Benefits:
- Instant (no purge needed)
- Old version still cached (if needed)
- Simple cache busting

Implementation:
- Append version query param: photo.jpg?v=2
- Or include in filename: photo-uuid.jpg (UUID changes on upload)
```

## Complete Upload/Download Flow

### Full Upload Flow with All Features

**Step 1: Client Initiates Upload**:
```
POST /api/files/upload-request
Body: {
    "filename": "vacation-video.mp4",
    "content_type": "video/mp4",
    "file_size": 524288000  // 500 MB
}
```

**Step 2: Server Validation & Presigned URL**:
```
Server:
1. Authenticate user
2. Check quota: user storage + file_size < limit
3. Validate file type: video/mp4 allowed
4. Validate size: < 1 GB limit

5. Generate file_id (UUID)
6. Generate S3 key: "uploads/{user_id}/{file_id}/vacation-video.mp4"

7. Generate presigned URL (1 hour expiration)
8. Save to database:
   - file_id, user_id, filename, status=pending, size

9. Return to client:
   {
       "file_id": "abc-123",
       "upload_url": "https://s3.../presigned...",
       "expires_at": "2025-01-15T11:00:00Z"
   }
```

**Step 3: Client Uploads to S3**:
```
Client:
- PUT request to upload_url
- Multipart upload for large files (>100 MB)
- Progress tracking
- Retry failed parts

S3 stores file
```

**Step 4: S3 Event Notification**:
```
S3 Event: ObjectCreated
→ Triggers Lambda/SNS
→ Backend receives notification

Backend:
UPDATE files SET status = 'uploaded' WHERE file_id = 'abc-123';

Optional:
- Trigger transcoding job
- Generate thumbnail
- Virus scan
- Update user quota
```

**Step 5: Post-Processing** (if needed):
```
For video:
- Transcode to multiple formats
- Generate HLS playlist
- Create thumbnail
- Extract metadata

For image:
- Generate thumbnails (small, medium, large)
- Compress
- Extract EXIF data

Update status: processing → ready
```

**Step 6: Client Polls for Completion**:
```
Client polls: GET /api/files/{file_id}/status

Response:
{
    "file_id": "abc-123",
    "status": "ready",
    "cdn_url": "https://cdn.example.com/files/abc-123/vacation-video.mp4",
    "thumbnail_url": "https://cdn.example.com/thumbs/abc-123.jpg"
}
```

## Advanced Features

### 1. Resumable Uploads

**Why Needed**:
```
Large files (1 GB+)
Upload can fail mid-way
Don't want to re-upload entire file
```

**Multipart Upload**:

**How It Works**:
```
1. Initiate multipart upload → Get upload_id
2. Split file into chunks (e.g., 10 MB each)
3. Upload each chunk independently with part number
4. Chunks can be uploaded in parallel
5. Complete multipart upload → S3 assembles file
```

**Implementation**:

**Step 1: Initiate**:
```
POST /api/files/{file_id}/multipart-init

Server:
upload_id = s3.create_multipart_upload(
    Bucket='bucket',
    Key='path/to/file.mp4'
)

Return: {
    "upload_id": "xyz",
    "part_size": 10485760  // 10 MB
}
```

**Step 2: Upload Parts**:
```
For each part:
POST /api/files/{file_id}/upload-part?part=1&upload_id=xyz

Server generates presigned URL for part upload:
presigned_url = s3.generate_presigned_url(
    'upload_part',
    Params={
        'Bucket': 'bucket',
        'Key': 'path/to/file.mp4',
        'PartNumber': 1,
        'UploadId': 'xyz'
    }
)

Client uploads part to presigned URL
S3 returns ETag for part
```

**Step 3: Complete**:
```
POST /api/files/{file_id}/multipart-complete?upload_id=xyz
Body: {
    "parts": [
        {"part_number": 1, "etag": "abc"},
        {"part_number": 2, "etag": "def"},
        ...
    ]
}

Server:
s3.complete_multipart_upload(
    Bucket='bucket',
    Key='path/to/file.mp4',
    UploadId='xyz',
    MultipartUpload={'Parts': parts}
)

S3 assembles parts into single file
```

**Resume Upload**:
```
If upload fails:
1. Client tracks which parts completed (ETags)
2. On resume, GET /api/files/{file_id}/parts?upload_id=xyz
3. Server returns list of completed parts
4. Client uploads only missing parts
```

**Benefits**:
- Upload can be paused and resumed
- Parallel upload of chunks (faster)
- Failed chunks can be retried independently
- Progress tracking per chunk

### 2. Progress Tracking

**Server-Side Tracking**:
```
Not practical - server not involved in upload

Use S3 event notifications for milestones:
- Upload started
- Upload completed
- Parts completed (multipart)
```

**Client-Side Tracking**:
```
JavaScript (browser):

xhr.upload.addEventListener('progress', (e) => {
    const percent = (e.loaded / e.total) * 100;
    updateProgressBar(percent);
});

fetch with streams:
- Can track bytes uploaded
- Update UI in real-time
```

**Server Provides**:
```
Estimated upload time based on file size:
{
    "file_size": 524288000,
    "estimated_seconds": 120,  // Based on typical upload speed
    "part_size": 10485760
}

Client can show estimated time remaining
```

---

### 3. Virus Scanning & Validation

**When to Scan**:
```
After upload completes, before making file accessible
```

**Implementation**:

**S3 Event → Lambda**:
```
S3 ObjectCreated event → Lambda function

Lambda:
1. Download file from S3 (or stream)
2. Run antivirus scan (ClamAV)
3. If infected:
   - Delete file from S3
   - Update database: status = 'rejected', reason = 'virus'
   - Notify user
4. If clean:
   - Update database: status = 'ready'
   - Make file accessible
```

**File Format Validation**:
```
Validate Content-Type matches actual file:
- Check magic bytes (file signature)
- Example: JPEG starts with FF D8 FF
- Prevents malicious file disguised as image

Reject if mismatch
```

## State Synchronization

### Challenge: Database vs Blob Storage Consistency

**Problem**:
```
Two sources of truth:
1. Database: File metadata (status, owner, size)
2. S3: Actual file

What if they get out of sync?
- Database says file exists, S3 doesn't have it
- S3 has file, database doesn't know about it
```

**Solutions**:

**1. Database as Source of Truth**:
```
Database record created first (status = pending)
Upload happens
S3 event confirms upload
Database updated (status = ready)

If upload fails:
- Database record remains (status = pending)
- Background job cleans up abandoned uploads
- Or scheduled task deletes pending > 24 hours old
```

**2. S3 Events for Reconciliation**:
```
Listen to S3 events:
- ObjectCreated → Create/update database record
- ObjectRemoved → Mark database record as deleted

Keeps database in sync with S3
```

**3. Periodic Reconciliation**:
```
Background job (daily):
1. Query database for all files
2. Check if file exists in S3
3. If database says exists but S3 doesn't:
   - Mark as missing in database
   - Alert for investigation

4. Query S3 for all files
5. Check if file in database
6. If orphaned file in S3 (not in database):
   - Delete from S3 (cleanup)
```

## Cost Optimization

### Storage Costs

**S3 Storage Tiers**:
```
S3 Standard: $0.023/GB/month (frequent access)
S3 IA (Infrequent Access): $0.0125/GB/month (monthly access)
S3 Glacier: $0.004/GB/month (archival)

Lifecycle policies:
- Files older than 90 days → Move to S3 IA
- Files older than 1 year → Move to Glacier
```

**Delete Old Files**:
```
Retention policy:
- User deletes file → Soft delete (mark in database)
- After 30 days → Permanently delete from S3

Free up storage
```

---

### Transfer Costs

**S3 to Internet**:
```
Egress: $0.09/GB (first 10 TB/month)

Problem: High cost for popular files downloaded many times
```

**CDN Reduces Costs**:
```
S3 → CloudFront: $0.02/GB
CloudFront → Internet: $0.085/GB

90% cache hit rate:
- Only 10% of requests go to S3
- Saves ~80% on egress costs
```

**Presigned URLs Avoid Server Transfer Costs**:
```
Direct S3 upload/download:
- No server bandwidth usage
- No server egress costs
- All costs on S3/CloudFront
```

## Example Use Cases

### Video Platform (YouTube-style)

```
Upload:
1. Client requests upload → Server returns presigned URL
2. Client uploads video to S3 (multipart if >100MB)
3. S3 event triggers transcoding (Lambda → MediaConvert)
4. Generate HLS playlist (multiple quality levels)
5. Generate thumbnail
6. Update database: status = ready

Download:
1. Client requests video → Server returns CDN URL
2. CDN serves HLS playlist and video chunks
3. Adaptive bitrate streaming (client selects quality)
4. 95% cache hit rate at CDN edge
```

### Photo Sharing (Instagram-style)

```
Upload:
1. Presigned URL for original photo
2. Client uploads to S3
3. Lambda generates thumbnails (small, medium, large)
4. Store thumbnails in S3
5. CDN caches all versions

Download:
1. Feed requests thumbnail → CDN URL (instant, cached)
2. Full-size view → CDN URL (may need origin fetch first time)
```

### Document Management

```
Upload:
1. Presigned URL with virus scanning
2. S3 event → Lambda runs ClamAV
3. If clean, status = ready; if infected, delete + notify

Download:
1. Signed CDN URL (private documents)
2. URL expires in 1 hour
3. Audit log in database (who accessed when)
```
